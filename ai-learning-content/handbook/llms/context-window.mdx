# Context Window

## Purpose

This handbook document explains context windows—the maximum amount of information an LLM can process in a single request. Understanding context windows is essential for designing production AI applications that work within model limitations while delivering optimal performance.

---

# What is a Context Window?

## Definition

The context window is the maximum number of tokens an LLM can process in a single request, including both input (prompt + context) and output (response).

## Analogy

Think of the context window as the model's "working memory":
- It can only "see" and process a limited amount of information at once
- Anything beyond the window is ignored
- The model cannot reference information outside its current context

## Formula

```
Context Window = Input Tokens + Output Tokens

Example:
- Model context window: 100K tokens
- Your prompt + context: 80K tokens
- Maximum response: 20K tokens (100K - 80K)
```

# Context Window Challenges

## Challenge 1: Information Overload

**Problem**: More context doesn't always mean better results

**Why**: 
- Models can struggle with very long contexts
- Attention may dilute across too much information
- Important details may get "lost" in noise

**Solution**: Provide only relevant, high-quality context

## Challenge 2: Cost

**Problem**: Larger contexts cost more

**Example**:
```
Input: 100K tokens at $3/1M = $0.30 per request
vs
Input: 10K tokens at $3/1M = $0.03 per request

10x context = 10x cost
```

## Challenge 3: Latency

**Problem**: Processing larger contexts takes longer

**Impact**:
- Slower response times
- Higher memory usage
- Increased computational cost

## Challenge 4: Context Window Limits

**Problem**: Requests exceeding the limit fail

**Example**:
```
Model: GPT-3.5 (16K context)
Your input: 20K tokens
Result: ❌ Error - context window exceeded
```

# Context Management Strategies

## Strategy 1: Prioritization

Rank information by importance:

```
Priority 1 (Always include):
- System prompt
- Current user message
- Critical instructions

Priority 2 (Include if space):
- Recent conversation
- Retrieved documents
- User preferences

Priority 3 (Include if space):
- Older conversation
- Additional context
- Supplementary info
```

## Strategy 2: Truncation

Remove less important information:

```
Full conversation: 100 messages (80K tokens)
Available budget: 50K tokens

Strategy:
- Keep last 20 messages (20K tokens)
- Summarize first 80 messages (5K tokens)
- Total: 25K tokens (saves 55K)
```

## Strategy 3: Summarization

Condense information:

```
Option A: Full history
- 50 messages
- 40K tokens

Option B: Summary
- "User asked about X, Y, Z"
- 500 tokens

Savings: 39.5K tokens (98.75% reduction)
```

## Strategy 4: Retrieval (RAG)

Instead of sending everything, retrieve only what's relevant:

```
Without RAG:
- Send entire 200-page documentation (100K tokens)

With RAG:
- Index documentation
- Search for relevant sections
- Send only top 5 sections (5K tokens)

Savings: 95K tokens (95% reduction)
```

## Strategy 5: Chunking

Split large requests into smaller ones:

```
Large request:
- Analyze 100 documents (150K tokens)
- ❌ Exceeds context window

Chunked approach:
- Analyze 10 documents at a time (15K tokens)
- ✅ Fits within context
- Combine results
```

# Context Engineering Principles

## 1. Relevance Over Quantity

```
❌ Bad: Send everything "just in case"
✅ Good: Send only what the model needs
```

## 2. Dynamic Context

```
❌ Bad: Same context for every request
✅ Good: Assemble context based on current task
```

## 3. Runtime Assembly

```
❌ Bad: Pre-build static context
✅ Good: Build context dynamically per request
```

## 4. Token Budget Awareness

```
❌ Bad: Ignore token limits until error
✅ Good: Monitor and optimize continuously
```

## 5. Modular Context Providers

```
❌ Bad: One monolithic context builder
✅ Good: Separate providers for different sources
```

# Long Context Best Practices

## When Long Contexts Help

✅ Analyzing long documents
✅ Maintaining extended conversations
✅ Comparing multiple sources
✅ Code review across large codebases
✅ Multi-document synthesis

## When Long Contexts Hurt

❌ Simple Q&A (wasteful)
❌ When most context is irrelevant
❌ When cost is a concern
❌ When speed is critical
❌ When quality degrades

## Optimal Long Context Usage

```
1. Start with small context
2. Add only what's necessary
3. Monitor quality vs context size
4. Use RAG for very large knowledge bases
5. Summarize when possible
```

# Context Window vs Model Capability

## Important Distinction

Larger context windows don't automatically mean better performance:

```
Model A: 200K context, struggles with information retrieval
Model B: 32K context, excellent at using provided context

Model B may perform better for many tasks despite smaller context
```

## The "Lost in the Middle" Problem

Research shows models can struggle with information in the middle of very long contexts:

```
Context: [A, B, C, D, E, F, G, H, I, J]
Query: "What is in position F?"

Models perform worse on middle information
than on beginning or end information
```

## Best Practice

```
✅ Place critical information at the beginning or end
✅ Use RAG to surface relevant information
✅ Test with your specific use case
✅ Don't assume larger context = better results
```

# Interview Questions

## Q: What is a context window?

**A**: The context window is the maximum number of tokens an LLM can process in a single request, including both input and output. It represents the model's "working memory" for that request.

## Q: How do you manage context window limits?

**A**: Strategies include: prioritization (send only important information), truncation (remove less important data), summarization (condense information), RAG (retrieve only relevant documents), and chunking (split large requests into smaller ones).

## Q: What's the "lost in the middle" problem?

**A**: Research shows that LLMs perform worse on information located in the middle of very long contexts compared to information at the beginning or end. This affects how we should structure context for optimal performance.

## Q: Does a larger context window always mean better performance?

**A**: No. While larger contexts enable more information to be included, they don't automatically improve performance. Models can struggle with very long contexts, and irrelevant information can degrade response quality. The key is providing the right context, not the most context.

## Q: How does context window size affect cost?

**A**: LLM providers charge per token. Larger contexts mean more input tokens, which increases cost. For example, a 100K token context costs 10x more than a 10K token context at the same per-token rate.
