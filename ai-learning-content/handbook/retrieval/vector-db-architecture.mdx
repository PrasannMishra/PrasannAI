```markdown
# Vector Database Architecture

## Purpose

This document explains the internal architecture of modern vector databases. You will learn about collections, segments, payloads, the Write-Ahead Log, mutable and immutable segments, and the background optimizer.

---

# Internal Storage Architecture

A vector database is more than an HNSW graph. Internally, several components work together:
```

Collection ├── Segment A │ ├── Vectors │ ├── Payload │ └── HNSW Index └── Segment B ├── Vectors ├── Payload └── HNSW Index

```javascript

Each component has a specific responsibility.

---

# Collections

A **Collection** is the highest-level logical container in a vector database, similar to a table in a relational database.

| Relational Database | Vector Database |
|---------------------|-----------------|
| Database | Database |
| Table | Collection |
| Row | Vector (Point) |

Every vector inside a collection shares the same configuration:
- Embedding dimension
- Similarity metric
- Indexing algorithm
- Optimization settings
```

```javascript
# Segments

Collections are not stored as one massive file. Instead, they are divided into **Segments**.
```

Collection ├── Segment 1 ├── Segment 2 ├── Segment 3 └── Segment 4

````javascript

A segment is an independent storage unit containing:
- Vectors
- Payload
- HNSW index
- Internal statistics

Each segment maintains its own HNSW graph.

## Why Segments Exist

Imagine storing fifty million vectors inside one graph. Problems emerge:
- Long optimization times
- High memory consumption
- Difficult maintenance
- Poor scalability

Segments solve this by partitioning the collection into manageable pieces. During search, every segment performs its own nearest-neighbor search, after which the database merges the results into a global ranking.

---

# Payload

An embedding alone has very little meaning. A **Payload** stores metadata associated with the vector:

```json
{
  "title": "Hierarchical Navigable Small World",
  "day": 12,
  "category": "roadmap",
  "difficulty": "Intermediate",
  "tags": ["vector-db", "hnsw"]
}
````

Payload is not used to compute semantic similarity. Instead, it enables:

- Filtering
- Authorization
- Categorization
- Result presentation

## Payload Filtering

Suppose the student asks: "Show beginner lessons about embeddings."

The query becomes:

```javascript
Semantic Search + category = roadmap + difficulty = Beginner
```

Only vectors matching the metadata participate in ranking. Combining semantic retrieval with structured filtering is one of the major strengths of vector databases.

---

# Write-Ahead Log (WAL)

Rebuilding an HNSW graph after every insertion would make writes slow and unreliable. Instead, every insertion first enters the __Write-Ahead Log (WAL)__.

```javascript
Insert → Write Ahead Log → Acknowledge Client → Background Processing
```

The WAL is an append-only log that guarantees durability. If the database crashes before optimization completes, it can replay the log during startup and recover the missing vectors.

__Without WAL:__

```javascript
Insert → Crash → Vector Lost
```

__With WAL:__

```javascript
Insert → Write WAL → Crash → Restart → Replay WAL → Recover Vector
```

The client receives a successful response immediately, while the database safely completes indexing in the background.

# Mutable and Immutable Segments

A vector database optimizes two competing operations: fast writes and fast reads.

## Mutable Segment

A __Mutable Segment__ accepts all new writes:

```javascript
Insert Vector → Mutable Segment
```

Characteristics:

- Accepts inserts and updates
- Immediately searchable
- Small in size
- Optimized for write performance

Contrary to common misconception, vectors inside the mutable segment are searchable immediately. The database performs a lightweight search over these recent vectors while continuing to use HNSW for larger optimized segments.

## Immutable Segment

After enough vectors accumulate, the mutable segment is converted into an optimized immutable segment:

```javascript
Mutable Segment → Optimization → Immutable Segment → HNSW Index
```

Immutable segments are optimized for retrieval rather than modification:

- Read optimized
- Indexed with HNSW
- Rarely modified
- Efficient memory layout

Most searches in production systems operate primarily on immutable segments.

---

# Background Optimizer

The __Optimizer__ is responsible for maintaining the health of the collection. It continuously performs maintenance tasks in the background:

- Building HNSW indexes
- Merging small segments
- Removing deleted vectors
- Compacting fragmented storage
- Improving memory layout
- Reducing search overhead

Because optimization runs asynchronously, user queries continue without interruption.

---

# Complete Insert Lifecycle

The lifecycle of a newly generated embedding:

```javascript
Document → Chunking → Embedding Model → Vector → Collection → 
Write-Ahead Log → Mutable Segment → Immediately Searchable → 
Background Optimizer → Immutable Segment → HNSW Index
```

__Important observations:__

- The application does not wait for optimization
- Newly inserted vectors remain searchable
- Index optimization occurs independently
- Search quality improves as optimization completes

This architecture provides both low write latency and high read performance.

---

# Query Execution During Optimization

Suppose a user inserts five new lessons and immediately performs a search. The database searches two sources simultaneously:

```javascript
Query
    ├── Optimized Segments (HNSW)
    └── Mutable Segment
        ↓
    Merge Results → Return Top-K
```

The user receives results that include recently inserted vectors even though they have not yet been incorporated into the main HNSW index.

# Engineering Considerations

- __Batch embeddings__ whenever possible instead of inserting one vector at a time
- __Avoid rebuilding collections__ unnecessarily
- __Monitor optimization activity__ for very large imports
- __Understand that indexing is eventually optimized__, not synchronously rebuilt
- __Design applications__ assuming recently inserted vectors are available for search almost immediately

---

# Key Takeaways

- A Collection is the logical container for vectors
- Segments partition collections for scalability
- Payload stores metadata for filtering and presentation
- The Write-Ahead Log guarantees durability
- Newly inserted vectors are immediately searchable through mutable segments
- The Optimizer runs asynchronously to improve retrieval performance
- A vector database retrieves relevant context; the LLM generates the answer
- Retrieval quality depends on the combined effectiveness of chunking, embeddings, indexing, and metadata
