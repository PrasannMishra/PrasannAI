# Vector Similarity Metrics

## Purpose

This document explains how to measure semantic similarity between vectors. You will learn why cosine similarity is the preferred metric for semantic search, how vector normalization works, and when to use different similarity metrics.

---

# Why We Need Similarity Metrics

If every document is represented as a vector, we need a way to answer: **Which vectors are closest?**

A similarity metric is a mathematical function that measures how closely two vectors are related. The vector with the highest similarity score is considered the most relevant.

---

# Common Similarity Metrics

| Metric | Measures | Common Usage |
|--------|----------|--------------|
| Euclidean Distance | Straight-line distance | Clustering |
| Manhattan Distance | Grid distance | Optimization |
| Dot Product | Magnitude × Direction | Recommendation Systems |
| Cosine Similarity | Direction | Semantic Search |

Modern RAG systems almost always use **Cosine Similarity**.

---

# Thinking of Vectors as Arrows

Instead of imagining vectors as lists of numbers, imagine them as arrows.
```

```javascript
        Y
        ▲
        │
        │
   B ↗
     /
    /
   /
```

O────────────────────► X\
\
\

```javascript

Each vector has two properties:
- **Direction** - represents the semantic concept
- **Length (Magnitude)** - represents how far the arrow extends

For semantic search, **direction is much more important than length**.

---

# Euclidean Distance

Euclidean Distance measures the straight-line distance between two vectors.
```

A ●──────────────● B

```javascript

The closer the points, the more similar they are assumed to be.

**Limitation:** Large vectors may appear far apart even when they represent the same semantic meaning.

**Example:**
```

A = [2,3] B = [4,6]

```javascript

Notice that B = 2 × A. They point in exactly the same direction, but Euclidean distance considers them different because the endpoints are farther apart.

---

# Cosine Similarity

Cosine Similarity compares the **angle** between two vectors.

Instead of asking: "How far apart are these vectors?"
it asks: "Are these vectors pointing in the same direction?"

This makes cosine similarity ideal for semantic search.

---

# Interpreting Cosine Similarity

Cosine similarity produces values between **-1 and 1**:

| Score | Interpretation |
|------:|----------------|
| 1.0 | Identical direction |
| 0.9 | Extremely similar |
| 0.7 | Related |
| 0.5 | Weak relationship |
| 0 | No relationship |
| -1 | Opposite direction |

Most document retrieval systems return vectors with scores close to **1**.

---

# Vector Magnitude

The length of a vector is called its **magnitude** (also known as its norm).

**Example:**
```

[3,4] Magnitude: √(3² + 4²) = 5

```javascript

A longer vector does not necessarily represent more meaningful information. Its length should not influence semantic similarity.

---

# Vector Normalization

Normalization solves the magnitude problem. Instead of comparing vectors with different magnitudes, every vector is scaled so that its magnitude becomes exactly **1**.

**Original vector:** `[3,4]` with magnitude `5`
**Normalized vector:** `[0.6,0.8]` with magnitude `1`

Only the direction remains unchanged.

**Why it matters:**
```

[2,3] and [4,6] have different magnitudes After normalization: [0.554,0.832] and [0.554,0.832] They become identical because they point in the same direction

```javascript

---

# Why Cosine Similarity Works Better

Imagine three vectors:
- A and B point in the same direction
- C points somewhere else

Even though B is much longer than A, they represent nearly the same meaning. Cosine similarity correctly identifies A and B as similar because their directions match.

---

# Production Best Practices

1. **Prefer Cosine Similarity** - Unless the model documentation recommends otherwise
2. **Normalize When Required** - Follow embedding model recommendations
3. **Use Same Model** - Generate document and query embeddings using the same model
4. **Retrieve Multiple Candidates** - Retrieve multiple chunks rather than a single result
5. **Combine with Metadata Filters** - Use filters for better precision

---

# Common Mistakes

- Mixing embeddings generated by different models
- Comparing vectors using an unsupported similarity metric
- Ignoring model-specific normalization requirements
- Using Euclidean distance for semantic search

---

# Key Takeaways

- Cosine similarity compares vector direction, not magnitude
- Vector normalization removes magnitude differences
- Cosine similarity is the preferred metric for semantic search
- Normalization improves retrieval quality
- Always use the same embedding model for indexing and querying
