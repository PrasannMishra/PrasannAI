# Prompt Templates

## Purpose

This handbook document covers prompt templates—the foundation of reusable, maintainable prompt systems in production AI applications. Templates separate prompt structure from dynamic content, enabling prompts to evolve independently from application logic.

---

# What are Prompt Templates?

## Definition

Prompt templates are reusable prompt structures with placeholders for dynamic values. They define the structure and wording of prompts while allowing specific values to be injected at runtime.

## Core Concept

Instead of hardcoding complete prompts, you create templates:

```
Template:
Role: {{role}}
Task: {{task}}
Context: {{context}}
Output: {{format}}

Runtime:
Role: Senior Engineer
Task: Review this code
Context: React project
Output: Markdown report
```

## Why Templates Matter

As AI applications grow, prompts become one of the largest pieces of business logic. Templates enable:
- **Reusability**: One template, many uses
- **Maintainability**: Update once, apply everywhere
- **Consistency**: Same structure across features
- **Testability**: Test template logic separately
- **Version control**: Track prompt evolution

# Template Syntax

## Common Patterns

### 1. Mustache-Style (Double Curly Braces)

```
Role: {{role}}
Task: {{task}}
Context: {{context}}
```

**Usage**: Most common, widely supported

### 2. Template Literals (JavaScript)

```javascript
`Role: ${role}
Task: ${task}
Context: ${context}`
```

**Usage**: Native JavaScript, no dependencies

### 3. Handlebars-Style

```
Role: {{role}}
{{#if experience}}
Experience: {{experience}}
{{/if}}
```

**Usage**: Conditional logic, loops

### 4. Custom Delimiters

```
Role: <%role%>
Task: <%task%>
```

**Usage**: Avoid conflicts with template engine syntax

# Template Variables

## Types of Variables

### 1. Required Variables

Must be provided or template fails:

```
Task: Review this {{language}} code for {{focus_area}}
```

Both `language` and `focus_area` are required.

### 2. Optional Variables

Have defaults or can be omitted:

```
{{#if include_examples}}
Examples:
{{examples}}
{{/if}}
```

### 3. Conditional Variables

Only included when condition is met:

```
{{#if is_security_review}}
Security Focus:
- OWASP Top 10
- Authentication
- Authorization
{{/if}}
```

### 4. List Variables

Iterate over arrays:

```
Files to review:
{{#each files}}
- {{this.name}} ({{this.lines}} lines)
{{/each}}
```

# Template Design Patterns

## Pattern 1: Simple Substitution

**Use case**: Basic variable replacement

```
Template:
You are a {{role}}. {{task}}

Example:
You are a Senior Engineer. Review this code.
```

## Pattern 2: Section-Based

**Use case**: Organized, multi-part prompts

```
Template:
Role:
{{role}}

Context:
{{context}}

Task:
{{task}}

Constraints:
{{constraints}}

Output Format:
{{output_format}}

Example:
Role:
Senior Frontend Engineer

Context:
React 18 project with TypeScript

Task:
Review this component

Constraints:
Focus on performance and accessibility

Output Format:
Markdown with sections
```

## Pattern 3: Conditional Sections

**Use case**: Optional prompt parts

```
Template:
{{#if include_examples}}
Examples:
{{examples}}

{{/if}}

{{#if include_constraints}}
Constraints:
{{constraints}}
{{/if}}

Task:
{{task}}
```

## Pattern 4: Composable Templates

**Use case**: Build complex prompts from simple parts

```
Base Template:
{{> header}}
{{> context}}
{{> task}}
{{> footer}}

Partials:
header: "You are a {{role}}."
context: "Context: {{context}}"
task: "Task: {{task}}"
footer: "Output: {{format}}"
```

## Pattern 5: Loop-Based

**Use case**: Process lists of items

```
Template:
Review each file:

{{#each files}}
File: {{this.name}}
{{#each this.issues}}
- Issue: {{this.description}}
  Severity: {{this.severity}}
{{/each}}

{{/each}}
```

# Template Variables Best Practices

## 1. Use Descriptive Names

❌ **Bad**:
```
{{x}}, {{y}}, {{z}}, {{a}}
```

✅ **Good**:
```
{{role}}, {{task}}, {{context}}, {{output_format}}
```

## 2. Provide Defaults

```typescript
const template = {
  role: 'Senior Engineer', // default
  task: required,
  context: 'General', // default
};
```

## 3. Validate Required Variables

```typescript
function render(template: string, variables: object) {
  const required = extractRequired(template);
  const missing = required.filter(key => !(key in variables));
  
  if (missing.length > 0) {
    throw new Error(`Missing variables: ${missing.join(', ')}`);
  }
  
  return renderTemplate(template, variables);
}
```

## 4. Document Variables

```yaml
# code-review.md
---
name: Code Review
description: Review code for quality, performance, and security
variables:
  - name: role
    required: true
    description: Reviewer expertise level
    example: "Senior Frontend Engineer"
  
  - name: focus_areas
    required: false
    default: ["bugs", "performance", "security"]
    description: Areas to focus on
---
```

# Template Examples

## Example 1: Code Review Template

```yaml
---
name: Code Review
version: 2.0.0
---

Role: {{role}}

Context:
{{project_context}}

Task:
Review the following code changes.

Focus Areas:
{{#each focus_areas}}
- {{this}}
{{/each}}

{{#if include_security}}
Security Checklist:
- OWASP Top 10 vulnerabilities
- Authentication/authorization issues
- Input validation
- Sensitive data exposure
{{/if}}

{{#if include_performance}}

- Unnecessary re-renders
- Memory leaks
- Bundle size impact
- Network requests
{{/if}}

Output Format:
## Summary
[2-3 sentences]

## Critical Issues
[Must fix before merge]

## Suggestions
[Nice-to-have improvements]

## Positive Notes
[What was done well]
```

## Example 2: Documentation Template

```yaml
---
name: API Documentation
version: 1.0.0
---

You are a technical writer creating documentation for {{target_audience}}.

Document the following {{resource_type}}:

{{#if description}}
## Description
{{description}}
{{/if}}

## Endpoint
{{method}} {{path}}

{{#if parameters}}
## Parameters
{{#each parameters}}
### {{this.name}}
- Type: {{this.type}}
- Required: {{this.required}}
- Description: {{this.description}}
{{/each}}
{{/if}}

{{#if request_body}}
## Request Body
{{request_body}}
{{/if}}

{{#if response_examples}}
## Examples
{{#each response_examples}}
### {{this.status}}
```json
{{this.body}}
```
{{/each}}
{{/if}}

## Error Responses
{{error_responses}}
```

# Template Composition

## Building Complex Prompts

Instead of one large template, compose from smaller parts:

```typescript
// Compose a code review prompt
const prompt = compose(
  baseRole('Senior Engineer'),
  projectContext('React app'),
  task('Review PR #123'),
  focusAreas(['performance', 'security']),
  outputFormat('markdown')
);

// Result:
// Role: Senior Engineer
// Context: React app
// Task: Review PR #123
// Focus: performance, security
// Output: markdown
```

## Template Inheritance

```yaml
# base-code-review.md
Role: {{role}}
Task: Review code

# security-review.md
extends: base-code-review.md
add:
  - Security focus: OWASP Top 10
  - Checklist: authentication, authorization, input validation
```

# Template Testing

## Test Cases

```typescript
interface TemplateTest {
  name: string;
  template: string;
  variables: object;
  expectedOutput: string;
  shouldFail?: boolean;
}

const tests: TemplateTest[] = [
  {
    name: 'Basic substitution',
    template: 'Hello {{name}}',
    variables: { name: 'World' },
    expectedOutput: 'Hello World'
  },
  {
    name: 'Missing required variable',
    template: 'Hello {{name}}',
    variables: {},
    expectedOutput: '',
    shouldFail: true
  },
  {
    name: 'Conditional section',
    template: '{{#if show}}Visible{{/if}}',
    variables: { show: true },
    expectedOutput: 'Visible'
  }
];
```

## Validation

```typescript
function validateTemplate(template: string) {
  const issues = [];
  
  // Check for undefined variables
  const variables = extractVariables(template);
  
  // Check for balanced delimiters
  const openCount = (template.match(/\{\{/g) || []).length;
  const closeCount = (template.match(/\}\}/g) || []).length;
  
  if (openCount !== closeCount) {
    issues.push('Unbalanced template delimiters');
  }
  
  return issues;
}
```

# Template Management

## Storage Options

### 1. File System

```
prompts/
├── chat/
│   └── general.md
└── review/
    └── code.md
```

**Pros**: Version control, easy editing
**Cons**: Requires file I/O

### 2. Database

```sql
CREATE TABLE prompts (
  id: string,
  name: string,
  category: string,
  content: text,
  version: string,
  created_at: timestamp
);
```

**Pros**: Dynamic updates, metadata
**Cons**: More complex

### 3. Code Constants

```typescript
const PROMPTS = {
  codeReview: `Role: {{role}}\nTask: Review code`,
  documentation: `Write docs for {{resource}}`
};
```

**Pros**: Type safety, easy deployment
**Cons**: Requires redeployment to update

# Template Evolution

## Versioning Strategy

```
v1.0.0 - Initial version
v1.1.0 - Added security checklist
v2.0.0 - Restructured sections, added examples
```

## Migration

When updating templates:
1. Keep old version available
2. Test new version thoroughly
3. Gradually migrate users
4. Monitor for issues
5. Deprecate old version

## A/B Testing

```typescript
// Test two prompt versions
const versionA = await testPrompt('v1.0.0', testCases);
const versionB = await testPrompt('v2.0.0', testCases);

// Compare metrics
console.log('Version A quality:', versionA.avgQuality);
console.log('Version B quality:', versionB.avgQuality);
```

# Common Template Mistakes

## ❌ Mistake 1: Too Many Variables

```
Template with 20+ variables
```

**Problem**: Hard to maintain, easy to break

**Fix**: Break into smaller templates

## ❌ Mistake 2: Unclear Variable Names

```
{{x1}}, {{data}}, {{temp}}
```

**Problem**: Unclear purpose

**Fix**: Use descriptive names

## ❌ Mistake 3: No Validation

```
Template renders without checking required variables
```

**Problem**: Runtime errors, incomplete prompts

**Fix**: Validate before rendering

## ❌ Mistake 4: Monolithic Templates

```
2000-word template trying to do everything
```

**Problem**: Hard to test, maintain, reuse

**Fix**: Compose from smaller templates

# Interview Questions

## Q: What are prompt templates?

**A**: Prompt templates are reusable prompt structures with placeholders for dynamic values. They separate prompt structure from content, enabling prompts to be versioned, tested, and reused across applications.

## Q: Why are prompt templates important?

**A**: Templates enable reusability (one template, many uses), maintainability (update once, apply everywhere), consistency (same structure across features), testability (test template logic separately), and version control (track prompt evolution).

## Q: What's the difference between a template and a hardcoded prompt?

**A**: A hardcoded prompt is complete and static. A template has placeholders for dynamic values that are filled at runtime. Templates enable reusability and maintainability; hardcoded prompts don't.

## Q: How do you organize prompt templates?

**A**: Organize by category (chat, review, generation), use consistent naming conventions, version templates, store in version control, and document variables and usage. Consider directory structure: `prompts/{category}/{purpose}-{variant}.md`.

## Q: What are common template variables?

**A**: Common variables include: role (expertise level), task (what to do), context (background information), constraints (limitations), output_format (response structure), and domain-specific variables like language, framework, or focus areas.
