# Prompt Engineering

## Purpose

This handbook document provides a comprehensive guide to prompt engineering—the discipline of designing instructions that guide Large Language Models toward producing accurate, useful, and consistent responses. In production systems, prompts are carefully designed software artifacts that evolve over time, are versioned, reviewed, tested, and reused across applications.

---

# What is Prompt Engineering?

## Definition

Prompt Engineering is the process of designing instructions that guide a Large Language Model toward producing accurate, useful, and consistent responses.

## Beyond "Asking Better Questions"

Many beginners think prompting is simply "asking better questions." In production systems, prompts are much more:

- **Software artifacts**: Stored in version control, reviewed, tested
- **Evolving**: Improved over time based on feedback
- **Reusable**: Shared across applications
- **Maintainable**: Structured for easy updates
- **Testable**: Evaluated for consistency and quality

## The Production Mindset

Treat prompts like production code:
- Version control
- Code review
- Testing
- Documentation
- Refactoring
- Monitoring

# Why Prompt Engineering Matters

## Impact on AI Applications

Prompt quality directly affects:

### Accuracy
- Well-designed prompts produce more accurate responses
- Clear instructions reduce hallucinations
- Specific guidance improves factual correctness

### Consistency
- Structured prompts produce predictable outputs
- Format specifications ensure uniform responses
- Reusable templates maintain consistency across features

### Reliability
- Tested prompts behave predictably
- Versioned prompts enable rollback
- Documented prompts aid debugging

### Cost
- Better prompts reduce retry rates
- Clearer instructions reduce token waste
- Efficient prompts lower API costs

### User Experience
- Consistent response format
- Appropriate tone and style
- Meeting user expectations

# Prompt Anatomy

## Standard Structure

Most production prompts contain several logical sections:

```
System Instructions
    ↓
Role
    ↓
Context
    ↓
Task
    ↓
Constraints
    ↓
Output Format
```

## Section Breakdown

### 1. System Instructions

**Purpose**: Define the assistant's long-term behavior and identity

**Examples**:
```
You are a senior software engineer specializing in React.
You are an AI code reviewer focused on security and performance.
You are a technical documentation expert.
```

**Characteristics**:
- Defines who the model is
- Usually static across conversations
- Sets the tone and expertise level
- Establishes boundaries

### 2. Role

**Purpose**: Assign specific expertise to the model

**Examples**:
```
You are a Staff Frontend Engineer with 10 years of experience.
You are a security expert specializing in OWASP vulnerabilities.
You are a data scientist focused on statistical analysis.
```

**Characteristics**:
- More specific than system instructions
- Can change per task
- Influences response style and depth
- Improves response quality

### 3. Context

**Purpose**: Provide background information the model needs

**Examples**:
```
Project: E-commerce platform built with React and TypeScript
Team: 5 frontend developers, junior to senior level
Standards: ESLint, Prettier, Jest for testing
```

**Characteristics**:
- Task-specific information
- Can be dynamic per request
- Helps model understand the situation
- Often provided by ContextBuilder

### 4. Task

**Purpose**: Define what the model should do

**Examples**:
```
Review this pull request for performance issues.
Explain how this code works.
Generate documentation for this function.
```

**Characteristics**:
- Clear, actionable instruction
- The core of the prompt
- Changes per interaction
- Should be specific

### 5. Constraints

**Purpose**: Set boundaries and requirements

**Examples**:
```
Focus only on React performance.
Do not suggest external libraries.
Maximum response length: 500 words.
Use TypeScript examples only.
```

**Characteristics**:
- Limits the scope
- Prevents unwanted behavior
- Ensures relevant responses
- Can include negative constraints ("do not")

### 6. Output Format

**Purpose**: Specify how the response should be structured

**Examples**:
```
Return a markdown report with the following sections:
- Summary
- Issues Found
- Recommendations
- Priority

Format as JSON with keys: summary, issues, recommendations.
```

**Characteristics**:
- Defines response structure
- Enables programmatic parsing
- Ensures consistency
- Can include examples

# Prompt Design Principles

## 1. Be Explicit

❌ **Bad**:
```
Review this code.
```

✅ **Good**:
```
Review this React component for:
1. Performance issues (unnecessary re-renders, memoization opportunities)
2. Accessibility violations (ARIA labels, keyboard navigation)
3. Security vulnerabilities (XSS, unsafe DOM manipulation)
4. Code quality (readability, maintainability)

Provide specific line numbers and actionable suggestions.
```

## 2. Define the Desired Outcome

❌ **Bad**:
```
Help me with this function.
```

✅ **Good**:
```
Refactor this function to:
- Reduce complexity from O(n²) to O(n)
- Add proper error handling
- Include JSDoc comments
- Follow our team's naming conventions
```

## 3. Provide Relevant Context

❌ **Bad**:
```
Is this code good?
```

✅ **Good**:
```
Review this React hook for a real-time data dashboard.
Context:
- Uses WebSocket for live updates
- Must handle connection drops gracefully
- Target: 60fps rendering
- Browser support: Chrome, Firefox, Safari (latest 2 versions)
```

## 4. Specify Expected Output Format

❌ **Bad**:
```
List the issues.
```

✅ **Good**:
```
Return issues in this format:
## Issue Title
**Severity**: Critical/High/Medium/Low
**Line**: [line number]
**Problem**: [description]
**Fix**: [suggested solution]
```

## 5. Avoid Unnecessary Information

❌ **Bad**:
```
You are an AI assistant. I need you to help me review some code. 
I'm a developer working on a project. The code is in React. 
Can you please look at it and tell me what you think? 
I want to make sure it's good quality and follows best practices. 
Here's the code: [code]
```

✅ **Good**:
```
Review this React component for performance and accessibility.
[code]
```

## 6. Keep Prompts Maintainable

❌ **Bad**:
```
You are a senior software engineer with 10 years of experience 
in React and TypeScript who specializes in performance optimization 
and accessibility and knows all the latest React hooks and patterns...
```

✅ **Good**:
```
Role: Senior React Engineer
Focus: Performance, Accessibility
[Rest of prompt]
```

Use templates and variables instead of long static text.

# Common Prompt Anti-Patterns

## ❌ Anti-Pattern 1: Mixing Unrelated Instructions

**Bad**:
```
Review this code for security.
Also, explain how it works.
And suggest a better variable name.
Also, check if it follows PEP 8.
```

**Why it's bad**: Unrelated tasks confuse the model and reduce quality

**Good**: Separate into focused prompts or clearly section them

## ❌ Anti-Pattern 2: Hardcoding User-Specific Values

**Bad**:
```
Review John's code from the e-commerce project.
```

**Why it's bad**: Not reusable, requires prompt changes

**Good**: Use variables
```
Review {{author}}'s code from {{project}}.
```

## ❌ Anti-Pattern 3: Embedding Application Logic in Prompts

**Bad**:
```
If the user is premium, focus on advanced features.
If the user is free, focus on basic features.
Also check their subscription status from the database.
```

**Why it's bad**: Prompts should not contain business logic

**Good**: Application provides context, prompt focuses on task

## ❌ Anti-Pattern 4: Monolithic Prompts

**Bad**: A 2000-word prompt trying to do everything

**Why it's bad**: Hard to maintain, test, and optimize

**Good**: Break into focused, composable prompts

## ❌ Anti-Pattern 5: Vague Instructions

**Bad**:
```
Make it better.
```

**Why it's bad**: Model doesn't know what "better" means

**Good**:
```
Improve performance by:
- Reducing unnecessary re-renders
- Adding memoization where appropriate
- Optimizing expensive calculations
```

# Real-World Example

## Before (Unstructured)

```
Review this PR and tell me everything.
```

**Problems**:
- Vague ("everything")
- No focus areas
- No output format
- Unpredictable results

## After (Structured)

```
Role:
Senior Frontend Engineer

Task:
Review the pull request.

Focus Areas:
1. Bugs - Logic errors, edge cases
2. Performance - Re-renders, memoization, bundle size
3. Accessibility - ARIA labels, keyboard navigation, screen readers
4. Security - XSS, unsafe practices, sensitive data exposure
5. Code Quality - Readability, maintainability, TypeScript usage

Context:
- React 18 with TypeScript
- Using Redux Toolkit for state management
- Target: Enterprise application

Output Format:
## Summary
[2-3 sentence overview]

## Critical Issues
[Issues that must be fixed before merge]

## Suggestions
[Nice-to-have improvements]

## Positive Notes
[What was done well]
```

**Improvements**:
- Clear role
- Specific focus areas
- Relevant context
- Structured output
- Predictable results

# Prompt Testing

## Why Test Prompts?

Prompts are code. They need testing.

## What to Test

1. **Consistency**: Same input → Same output
2. **Accuracy**: Factual correctness
3. **Format**: Output matches specification
4. **Edge cases**: Unusual inputs
5. **Token usage**: Within budget
6. **Latency**: Acceptable response time

## Testing Methodology

```typescript
interface PromptTest {
  name: string;
  input: string;
  expectedFormat: string;
  expectedContent: string[];
  maxTokens: number;
}

const tests: PromptTest[] = [
  {
    name: 'PR review with bugs',
    input: 'Review PR #123 with bug fixes',
    expectedFormat: 'markdown',
    expectedContent: ['Critical Issues', 'Bugs'],
    maxTokens: 2000
  },
  // More test cases...
];

// Run tests
for (const test of tests) {
  const response = await aiCore.generate({
    prompt: buildPrompt(test.input)
  });
  
  // Assertions
  console.assert(response.tokens < test.maxTokens);
  console.assert(test.expectedContent.every(c => response.text.includes(c)));
}
```

# Prompt Versioning

## Why Version Prompts?

- Track changes over time
- Understand why behavior changed
- Rollback problematic changes
- A/B test different versions
- Debug issues

## Versioning Strategy

```
prompts/
├── pr-review/
│   ├── v1.0.0.md
│   ├── v1.1.0.md
│   ├── v2.0.0.md
│   └── latest.md → v2.0.0.md
```

## Version Documentation

Each version should include:
- Version number
- Date
- Author
- Changes from previous version
- Performance metrics
- Rollback instructions

# Best Practices

## 1. Use Templates

Store prompts as templates with placeholders:
```yaml
Role: {{role}}
Task: {{task}}
Context: {{context}}
```

## 2. Separate Concerns

- System prompts in one file
- Role prompts in another
- Task-specific prompts separate

## 3. Document Everything

- Why the prompt is designed this way
- What each section does
- Known limitations
- Expected behavior

## 4. Test Thoroughly

- Multiple scenarios
- Edge cases
- Different inputs
- Output validation

## 5. Review Like Code

- Code review for prompt changes
- A/B testing for significant changes
- Monitor production behavior

## 6. Monitor Performance

- Track response quality
- Monitor token usage
- Log failures
- Collect user feedback

# Interview Questions

## Q: What is prompt engineering?

**A**: Prompt engineering is the discipline of designing instructions that guide LLMs to produce accurate, useful, and consistent responses. In production, prompts are treated as software artifacts—versioned, tested, reviewed, and maintained.

## Q: What are the key components of a production prompt?

**A**: A production prompt typically includes: system instructions (define behavior), role (assign expertise), context (provide background), task (specify action), constraints (set boundaries), and output format (define structure).

## Q: Why is prompt structure important?

**A**: Structure improves consistency, maintainability, and testability. Well-structured prompts are easier to understand, modify, and debug. They produce more predictable outputs and enable better version control.

## Q: How do you test prompts?

**A**: Test prompts like code: create test cases with expected inputs and outputs, validate format and content, check token usage, test edge cases, and monitor production behavior. Use automated testing where possible.

## Q: What makes a good prompt?

**A**: Good prompts are: clear (unambiguous), specific (detailed requirements), structured (organized sections), reusable (template-based), testable (predictable output), and version-controlled (tracked in code).
