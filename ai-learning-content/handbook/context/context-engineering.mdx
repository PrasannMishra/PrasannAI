# Context Engineering

## Purpose

This handbook document covers Context Engineering—the discipline of selecting, organizing, and delivering the information that an LLM needs to perform a task effectively. While Prompt Engineering focuses on *how we ask*, Context Engineering focuses on *what the model knows at inference time*.

---

# What is Context Engineering?

## Definition

Context Engineering is the discipline of selecting, organizing, and delivering the information that an LLM needs to perform a task effectively.

## Core Principle

> **Context quality often matters more than prompt quality.**

A well-written prompt with poor context will produce worse results than a mediocre prompt with excellent context.

## Prompt vs Context

Many beginners treat these terms as interchangeable, but they serve different purposes:

| Prompt | Context |
|--------|---------|
| Instructions | Information |
| Defines the task | Supplies knowledge |
| Usually short | Can be very large |
| Mostly static | Highly dynamic |
| Created by developers | Built at runtime |

**Prompt tells the model what to do.**
**Context tells the model what it needs to know.**

---

# Why Context Matters

## The Power of Context

Consider two scenarios:

### Scenario 1: Prompt Only
```
Review this pull request.
```

**Model knows**: Nothing about the PR, codebase, or standards

**Result**: Generic, unhelpful feedback

### Scenario 2: Prompt + Rich Context
```
Review this pull request.

Context:
- PR #123: Add user authentication
- Files changed: 5 files, 234 lines
- Team standards: ESLint, TypeScript strict mode
- Architecture: React 18, Redux Toolkit
- Security requirements: OWASP compliance
- Previous reviews: 2 similar PRs with patterns
```

**Model knows**: Exactly what to review and against what standards

**Result**: Specific, actionable, high-quality feedback

## Real-World Impact

```
Same prompt, different context:

❌ Poor context (10% quality):
"Review this code"
Result: Generic feedback, misses important issues

✅ Rich context (90% quality):
"Review this React authentication component 
for security vulnerabilities, considering OWASP 
Top 10 and our team's security standards"
Result: Specific security issues, actionable fixes
```

# Context Assembly

## What is Context Assembly?

Context assembly is the process of gathering information from multiple sources and combining it into a coherent context for the LLM.

## The ContextBuilder Pattern

Just as PromptBuilder constructs prompts, ContextBuilder assembles the complete context:

```
ContextBuilder Responsibilities:
- Load conversation history
- Retrieve relevant documents
- Include user preferences
- Add project metadata
- Enforce context limits
- Prioritize important information
```

## Context Assembly Flow

```
User Request
    ↓
ContextBuilder
    ↓
┌─────────────────────────────────────┐
│ Context Sources:                    │
│ - Conversation History              │
│ - Retrieved Documents (RAG)         │
│ - User Profile                      │
│ - Application State                 │
│ - System Information                │
└─────────────────────────────────────┘
    ↓
Token Budget Management
    ↓
Prioritization & Truncation
    ↓
Final Context
    ↓
LLM
```

# Context Engineering Principles

## Principle 1: Relevance Over Quantity

```
❌ Bad: Send everything "just in case"
✅ Good: Send only what the model needs
```

**Why**: More context costs more, slows inference, and can dilute attention.

## Principle 2: Dynamic Context

```
❌ Bad: Same context for every request
✅ Good: Assemble context based on current task
```

**Why**: Different tasks need different context.

## Principle 3: Runtime Assembly

```
❌ Bad: Pre-build static context
✅ Good: Build context dynamically per request
```

**Why**: Context depends on user input and current state.

## Principle 4: Token Budget Awareness

```
❌ Bad: Ignore token limits until error
✅ Good: Monitor and optimize continuously
```

**Why**: Context window limits are real constraints.

## Principle 5: Modular Context Providers

```
❌ Bad: One monolithic context builder
✅ Good: Separate providers for different sources
```

**Why**: Modularity enables testing, reuse, and optimization.

# Context Management Strategies

## Strategy 1: Prioritization

Rank information by importance:

```
Priority 1 (Always include):
- System prompt
- Current user message
- Critical instructions

Priority 2 (Include if space):
- Recent conversation
- Retrieved documents
- User preferences

Priority 3 (Include if space):
- Older conversation
- Additional context
- Supplementary info
```

## Strategy 2: Truncation

Remove less important information:

```
Full conversation: 100 messages (80K tokens)
Available budget: 50K tokens

Strategy:
- Keep last 20 messages (20K tokens)
- Summarize first 80 messages (5K tokens)
- Total: 25K tokens (saves 55K)
```

## Strategy 3: Summarization

Condense information:

```
Option A: Full history
- 50 messages
- 40K tokens

Option B: Summary
- "User asked about X, Y, Z"
- 500 tokens

Savings: 39.5K tokens (98.75% reduction)
```

## Strategy 4: Retrieval (RAG)

Instead of sending everything, retrieve only what's relevant:

```
Without RAG:
- Send entire 200-page documentation (100K tokens)

With RAG:
- Index documentation
- Search for relevant sections
- Send only top 5 sections (5K tokens)

Savings: 95K tokens (95% reduction)
```

## Strategy 5: Chunking

Split large requests into smaller ones:

```
Large request:
- Analyze 100 documents (150K tokens)
- ❌ Exceeds context window

Chunked approach:
- Analyze 10 documents at a time (15K tokens)
- ✅ Fits within context
- Combine results
```

# Context Sources in Detail

## Conversation History

### What to Include
- Recent messages (last 5-10)
- First message (often contains important context)
- Key decisions or conclusions
- User preferences stated

### What to Exclude
- Very old messages
- Failed attempts
- Redundant information
- Off-topic tangents

### Management
```
Full history: 100 messages (80K tokens)
Budget: 50K tokens

Strategy:
1. Keep last 20 messages (20K tokens)
2. Summarize messages 21-100 (5K tokens)
3. Total: 25K tokens
```

## Retrieved Documents (RAG)

### Retrieval Process
1. Index all documents
2. Embed user query
3. Search for similar documents
4. Rank by relevance
5. Select top N documents
6. Add to context

### Best Practices
- Retrieve only relevant chunks (5-10 typically)
- Include metadata (source, date, relevance score)
- Limit total retrieved tokens (10-20K typical)
- Use semantic search, not just keywords

## User Profile

### What to Include
- Skill level (beginner/intermediate/advanced)
- Preferences (verbose/concise, formal/casual)
- Goals (what they're trying to achieve)
- Past interactions (what they've learned)
- Stated constraints (time, format, etc.)

### Dynamic vs Static
- Static: Skill level, preferences (rarely change)
- Dynamic: Current goal, recent interactions (change per session)

## Application State

### Relevant State
- Current view/page
- Selected items
- Filter/sort criteria
- User actions leading to this point
- Feature flags

### Irrelevant State
- Internal implementation details
- UI state not related to task
- Cached data not needed for this request

# Context Window Constraints

## The Budget Problem

Every request must fit within the model's context window:

```
Context Window: 100K tokens

Allocation:
- System prompt: 500 tokens
- Conversation history: 20K tokens
- Retrieved documents: 50K tokens
- Current message: 1K tokens
- Reserved for response: 28.5K tokens

Total: 100K tokens (exactly fits)
```

## Managing the Budget

### Monitoring
```typescript
interface ContextBudget {
  total: number;        // 100K
  used: number;         // Current usage
  available: number;    // total - used
  reserved: number;     // For response
}
```

### Warnings
```
⚠️ Warning at 80% usage
❌ Error at 95% usage
```

### Optimization
When approaching limits:
1. Reduce conversation history
2. Summarize instead of full text
3. Retrieve fewer documents
4. Truncate less important context
5. Ask user to simplify request

# Context Quality vs Quantity

## The Quality Principle

```
10K tokens of highly relevant context
> 100K tokens of mixed relevant/irrelevant context
```

## Measuring Quality

### Relevance Score
```
For each context piece, ask:
- How directly does this help answer the question?
- Is this information the model doesn't already know?
- Will this improve accuracy or just add noise?

Score: 0-10
Include if score > 7
```

### Impact Testing
```
Test with different context:
A: Full context (100K tokens) → 75% accuracy
B: Top 20% relevant (20K tokens) → 85% accuracy
C: Top 10% relevant (10K tokens) → 82% accuracy

Conclusion: More isn't better. Relevance matters most.
```

Continuous improvement:

```
1. Log context usage per request
2. Identify high-usage scenarios
3. Optimize context assembly
4. A/B test different strategies
5. Monitor quality metrics
6. Iterate based on data
```

## Best Practices

### Do's
✅ Prioritize recent and relevant information
✅ Use RAG for large knowledge bases
✅ Summarize when possible
✅ Monitor token usage
✅ Test context quality

### Don'ts
❌ Send entire databases
❌ Include every previous message
❌ Ignore token limits
❌ Use one-size-fits-all context
❌ Embed context logic in UI

# Context Engineering vs Prompt Engineering

## Complementary Disciplines

```
Prompt Engineering: How we ask
Context Engineering: What the model knows

Both are required for production AI:
- Great prompt + poor context = mediocre results
- Poor prompt + great context = mediocre results
- Great prompt + great context = excellent results
```

## Example

### Prompt Only
```
Write a blog post about React hooks.
```

### Context Only
```
React documentation, hooks guide, best practices...
```

### Prompt + Context
```
You are a technical blogger specializing in React.

Write a blog post about React hooks for intermediate developers.

Context:
- React 18 features
- Common hooks: useState, useEffect, useContext
- Best practices from React team
- Common mistakes to avoid
- Code examples from production apps

Audience: Developers with 6+ months React experience
Tone: Friendly, practical, example-driven
Length: 1500 words
```

**Result**: Targeted, informed, high-quality content

# Common Context Anti-Patterns

## ❌ Anti-Pattern 1: Context Overflow

```
Sending 150K tokens when context window is 100K
```

**Solution**: Implement token counting and truncation

## ❌ Anti-Pattern 2: Irrelevant Context

```
Including entire user history when only recent context matters
```

**Solution**: Prioritize and filter

## ❌ Anti-Pattern 3: Static Context

```
Same context for every user, every request
```

**Solution**: Dynamic context assembly

## ❌ Anti-Pattern 4: Context in UI

```
UI components assembling context
```

**Solution**: Centralize in ContextBuilder

## ❌ Anti-Pattern 5: No Optimization

```
Never reviewing or improving context strategy
```

**Solution**: Monitor, measure, iterate

# Interview Questions

## Q: What is context engineering?

**A**: Context engineering is the discipline of selecting, organizing, and delivering the information that an LLM needs to perform a task effectively. While prompt engineering focuses on how we ask, context engineering focuses on what the model knows at inference time.

## Q: Why is context often more important than prompts?

**A**: A well-written prompt with poor context produces worse results than a mediocre prompt with excellent context. Context provides the knowledge and information the model needs to generate accurate, relevant responses. Prompt quality affects how the model uses context; context quality determines what the model can possibly say.

## Q: What's the difference between a prompt and context?

**A**: A prompt is the instruction telling the model what to do. Context is the information the model needs to know to do it. Prompts are usually short and static; context can be very large and dynamic. Prompts define the task; context supplies the knowledge.

## Q: How do you manage context window limits?

**A**: Strategies include: prioritization (send only important information), truncation (remove less important data), summarization (condense information), RAG (retrieve only relevant documents), and chunking (split large requests). Monitor token usage and optimize continuously.

## Q: What is ContextBuilder and why is it important?

**A**: ContextBuilder is a component responsible for assembling context from multiple sources (conversation history, documents, user profile, etc.). It's important because it centralizes context logic, ensures consistent context quality, manages token budgets, and keeps context assembly separate from business logic and UI components.
