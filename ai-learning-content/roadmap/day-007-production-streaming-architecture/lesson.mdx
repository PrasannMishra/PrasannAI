# Day 7 — Production Streaming Architecture

---

# Overview

Yesterday we learned **why** streaming exists.

Today we answer a much harder question:

> **How does streaming actually work inside a production AI application?**

Modern AI systems involve multiple independent layers that cooperate to deliver a streamed response.

Understanding these layers is essential for building scalable and provider-independent AI applications.

---

# Production Streaming Flow

```mermaid
flowchart LR

User

↓

React UI

↓

AI SDK

↓

AI Backend

↓

Provider Adapter

↓

Claude/OpenAI/Ollama

↓

Stream Events

↓

Frontend Rendering
```

Every layer performs a different responsibility.

No layer should assume another layer's implementation details.

---

# Why Streaming Is More Than "Showing Text"

Streaming affects:

- networking
- UI rendering
- cancellation
- error handling
- provider abstraction
- conversation state
- token accounting
- application architecture

It is a cross-cutting concern that spans the entire stack.

---

# Backend Responsibilities

The backend is responsible for:

- Authentication
- Calling AI providers
- Provider abstraction
- Prompt assembly
- Context assembly
- Stream normalization
- Security
- Rate limiting
- Usage tracking

The backend **should not** contain UI logic.

---

# Frontend Responsibilities

The frontend is responsible for:

- Reading streamed data
- Updating the UI
- Rendering Markdown
- Displaying typing indicators
- Managing scroll behavior
- Handling cancellation
- Preserving partial responses

The frontend should remain provider-independent.

---

# Provider Abstraction

Every provider streams data differently.

Examples:

OpenAI

```text
data: {...}
```

Anthropic SDK

```ts
for await (const event of stream)
```

Ollama

```json
{"response":"Hello"}
```

Applications should never depend on these differences.

Instead, introduce a provider abstraction.

```mermaid
flowchart LR

Claude

OpenAI

Ollama

↓

Provider Adapter

↓

Unified Stream

↓

AI Backend
```

Changing providers should not require UI changes.

---

# Stream Normalization

The backend converts provider-specific formats into a common internal representation.

Instead of exposing raw provider events:

```text
Claude Event

↓

Provider Adapter

↓

StreamEvent

↓

Frontend
```

The frontend always consumes the same event model.

---

# Why the Frontend Still Parses Streams

A common misconception is:

> "The backend already parsed the provider stream, so the frontend should receive JavaScript objects."

This is impossible.

HTTP transports **bytes**, not JavaScript objects.

Therefore the browser always receives:

```
ReadableStream<Uint8Array>
```

The frontend must:

1. Read bytes.
2. Decode UTF-8.
3. Reconstruct events.
4. Parse JSON.
5. Update the UI.

Backend normalization and frontend parsing solve completely different problems.

---

# ReadableStream

The Fetch API exposes streamed responses through:

```ts
ReadableStream<Uint8Array>
```

Notice:

The browser never returns strings.

Only bytes.

These bytes must be decoded before use.

---

# TextDecoder

The browser uses:

```ts
new TextDecoder()
```

to convert UTF-8 bytes into text.

Pipeline:

```text
Bytes

↓

Uint8Array

↓

TextDecoder

↓

Text

↓

JSON

↓

Application
```

---

# Buffer Management

Streaming data may arrive in incomplete chunks.

Example:

Packet 1

```text
{"response":"Hel
```

Packet 2

```text
lo"}
```

Attempting to parse Packet 1 immediately will fail.

Correct approach:

```text
buffer += decodedChunk

↓

Split complete messages

↓

Keep incomplete fragment

↓

Wait for next chunk
```

Buffer management is mandatory in production streaming implementations.

---

# Server-Sent Events (SSE)

The backend sends events using the SSE protocol.

Example:

```text
data: {"text":"Hello"}

data: {"text":"World"}
```

Each event ends with:

```text
\n\n
```

The frontend reconstructs complete events before parsing them.

---

# Async Generators

Provider SDKs often expose streams using:

```ts
for await (const chunk of stream)
```

Async generators simplify streaming because each iteration already represents one logical event.

Provider adapters should expose async generators regardless of the underlying provider.

---

# AbortController

Users should always be able to stop generation.

```text
User

↓

Stop

↓

AbortController.abort()

↓

Request Cancelled
```

Partial output must remain visible.

Never discard already generated content.

---

# Continue Generation

Stopping generation destroys the model's inference state.

Therefore "Continue" does **not** resume inference.

Instead:

```text
Conversation

+

Partial Response

+

Continue Instruction

↓

New Request
```

The application asks the model to continue naturally from the previous response.

---

# Streaming State Machine

Messages should maintain explicit status.

```text
streaming

↓

completed

↓

stopped

↓

error
```

Avoid representing state using booleans.

Enums make the lifecycle easier to extend.

---

# Recommended StreamEvent Model

Instead of yielding plain text:

```ts
yield text
```

Prefer:

```ts
StreamEvent
```

Possible event types:

- chunk
- done
- error
- metadata

Future metadata may include:

- latency
- token usage
- finish reason
- model name

This design scales better as applications evolve.

---

# Conversation Configuration

Model settings belong to the conversation.

Example:

```text
Conversation

Provider

Model

Temperature

Max Tokens

System Prompt
```

Individual messages should not duplicate configuration.

---

# AI Chat UX Best Practices

Production chat applications should support:

- Streaming
- Stop Generation
- Continue Generation
- Regenerate
- Copy
- Copy Code
- Markdown
- Syntax Highlighting
- Auto Scroll
- Conversation History
- Token Usage
- Error Recovery

Good UX is just as important as correct networking.

---

# Best Practices

- Normalize providers in the backend.
- Keep the frontend provider-independent.
- Preserve partial responses.
- Use async generators.
- Centralize stream parsing.
- Model message lifecycle explicitly.

# Key Takeaways

- HTTP always transports bytes.
- Backend normalization and frontend parsing solve different problems.
- ReadableStream returns bytes, not strings.
- TextDecoder converts bytes into text.
- Buffer management is essential.
- Provider abstraction keeps applications maintainable.
- StreamEvent models scale better than raw text.
- Streaming UX requires careful state management.
