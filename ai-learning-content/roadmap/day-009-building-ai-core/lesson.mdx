# Day 9 — Building AI Core

---

# Overview

Up until now we've learned individual architectural pieces:

- PromptBuilder
- ContextBuilder
- Streaming
- AI SDK
- AI Backend
- Provider Adapters

Today's lesson answers an important question:

> **Where should all these components actually live?**

The answer is **AI Core**.

AI Core is not a UI library.

It is not a backend.

It is not tied to React, NestJS, or FastAPI.

It is a reusable framework responsible for orchestrating all AI-related workflows.

Every AI application we build in this roadmap will rely on AI Core.

---

# Why AI Core Exists

Imagine building three different products.

```
AI Chat

PR Reviewer

Learning Portal
```

Without AI Core, each project duplicates:

- Prompt construction
- Context assembly
- Provider integration
- Streaming
- Conversation handling

Maintenance quickly becomes difficult.

Instead:

```
AI Chat

↓

AI Core

↑

Learning Portal

↑

PR Reviewer
```

All applications share the same AI intelligence.

---

# Responsibilities of AI Core

AI Core should own every concern directly related to AI orchestration.

Responsibilities include:

- PromptBuilder
- ContextBuilder
- Provider Registry
- Conversation Manager
- Stream Manager
- Memory Manager (future)
- Tool Registry (future)
- Workflow Engine (future)
- RAG Pipeline (future)

It should never contain UI-specific logic.

---


Each module has a single responsibility.

---

# Module: PromptBuilder

PromptBuilder creates reusable prompts from templates.

Responsibilities:

- Template rendering
- Variable substitution
- Prompt versioning
- Validation

Example:

```yaml
Role:
{{role}}

Task:
{{task}}

Output:
{{format}}
```

Applications never hardcode prompts.

---

# Module: ContextBuilder

ContextBuilder gathers information required by the model.

Possible sources:

- Conversation history
- Retrieved documents
- User profile
- Project metadata
- Uploaded files
- Application state

It returns one optimized context object.

---

# Module: Provider Registry

Instead of directly instantiating providers:

```ts
new AnthropicProvider()
```

AI Core uses a registry.

```
Registry

↓

Claude

OpenAI

Ollama

Gemini
```

Benefits:

- Easy provider replacement.
- Plugin architecture.
- Central configuration.

---

# Module: Conversation Manager

Responsible for:

- Creating conversations.
- Managing message history.
- Conversation configuration.
- Token accounting.
- Metadata.

Conversation state belongs here, not in the UI.

---

# Module: Stream Manager

Provides a provider-independent streaming interface.

Responsibilities:

- Stream lifecycle.
- Event normalization.
- Cancellation.
- Finish reasons.
- Error propagation.

The UI consumes normalized stream events without caring which provider generated them.

---

# AI Core Interfaces

Every module should expose clear interfaces.

Example:

```ts
interface PromptBuilder {
    build(template, variables)
}

interface ContextBuilder {
    build(request)
}

interface Provider {
    generate()
    stream()
}
```

Implementation details remain hidden.

---

# AI Core Should Be Framework Agnostic

Avoid dependencies on:

- React
- NestJS
- Express
- FastAPI

AI Core should be usable from:

- Node.js
- Browser
- CLI
- Background workers
- Tests

This keeps it reusable.

---

# Extension Points

AI Core should grow through extension rather than modification.

Future modules:

```
AI Core

↓

Memory

↓

RAG

↓

Agents

↓

Tool Calling

↓

MCP

↓

Workflow Engine
```

The existing architecture remains stable.

---

# Dependency Flow

Dependencies should always point inward.

```text
React

↓

AI SDK

↓

Backend

↓

AI Core

↓

Provider
```

Never reverse this dependency direction.

# Design Principles

- Single Responsibility
- Dependency Inversion
- Open/Closed Principle
- Composition over Inheritance
- Provider Independence
- Interface-Driven Design

---

# Common Mistakes

❌ Putting PromptBuilder inside React.

❌ Provider-specific code in business logic.

❌ Separate AI implementations per application.

❌ Tight coupling between UI and providers.

❌ Hardcoded prompts.

# Key Takeaways

- AI Core is the reusable intelligence layer.
- Applications should depend on AI Core rather than providers.
- AI Core owns orchestration.
- Modules should have single responsibilities.
- Extension points allow the platform to evolve.

Future lessons will progressively expand AI Core while preserving its public interfaces, demonstrating how a well-designed framework can evolve without breaking existing applications.
