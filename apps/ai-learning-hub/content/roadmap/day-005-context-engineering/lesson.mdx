# Day 5 — Context Engineering

---

# Overview

One of the biggest misconceptions in AI development is that better prompts automatically produce better results.

In reality, modern AI systems succeed because they provide the model with the **right context**, not merely a well-written prompt.

Context Engineering is the discipline of selecting, organizing, and delivering the information that an LLM needs to perform a task effectively.

While Prompt Engineering focuses on *how we ask*, Context Engineering focuses on *what the model knows at inference time*.

---

# Learning Objectives

By the end of today's lesson, you should be able to:

- Explain what Context Engineering is.
- Differentiate prompts from context.
- Identify common context sources.
- Understand the role of ContextBuilder.
- Recognize why context quality is often more important than prompt quality.

---

# Prompt vs Context

Many beginners treat these terms as interchangeable, but they serve different purposes.

| Prompt | Context |
|---------|----------|
| Instructions | Information |
| Defines the task | Supplies knowledge |
| Usually short | Can be very large |
| Mostly static | Highly dynamic |
| Created by developers | Built at runtime |

A prompt tells the model **what to do**.

Context tells the model **what it needs to know**.

---

Instead, it is composed from several sources:

The model receives the assembled result as one request, but internally that request has been built from many components.

---

# What is Context?

Context includes any information that helps the model generate a better response.

Examples:

- Previous conversation
- User preferences
- Uploaded files
- Retrieved documentation
- Current project
- Application settings
- Code snippets
- Database results
- Tool outputs

The quality of the context often has a greater impact on the response than the wording of the prompt itself.

---

# Why Context Matters

Imagine asking:

```text
Review this Pull Request.
```

Without context, the model knows almost nothing.

Now imagine providing:

- PR title
- Changed files
- Team coding standards
- Architecture guidelines
- Linked Jira ticket
- Existing comments
- Repository structure

The response becomes dramatically more useful.

---

# ContextBuilder

Just as PromptBuilder constructs prompts, ContextBuilder assembles the complete context required by the model.

Responsibilities include:

- Loading conversation history.
- Retrieving relevant documents.
- Including user preferences.
- Adding project metadata.
- Enforcing context limits.
- Prioritizing important information.

Applications should request a completed context rather than assembling it manually.

# Context Sources

A mature AI application may gather information from many places.

Examples include:

- Conversation history
- Retrieval-Augmented Generation (RAG)
- Database records
- User profile
- Tool results
- Application configuration
- Uploaded documents
- External APIs

Not every request requires every source. ContextBuilder selects only what is relevant.

---

# The Cost of Context

Context is not free.

Every additional token increases:

- Cost
- Latency
- Memory usage

More context does not automatically improve quality.

The goal is to provide the **most relevant** context, not the largest amount of context.

---

# Context Window Constraints

Every LLM has a maximum context window.

When assembling context, ContextBuilder must ensure that:

- Important information is preserved.
- Irrelevant details are removed.
- Token limits are respected.

This balancing act becomes increasingly important as applications scale.

---

# Context Engineering Principles

1. Relevance over quantity.
2. Dynamic context beats static context.
3. Assemble context at runtime.
4. Separate context generation from business logic.
5. Measure token usage.
6. Prefer modular context providers.

These principles form the basis of production AI architectures.

---

# Common Anti-Patterns

❌ Sending the entire database.

❌ Including every previous conversation message.

❌ Ignoring token limits.

❌ Embedding context logic inside UI components.

❌ Hardcoding context assembly.

Instead, centralize context creation through ContextBuilder.

---

# Real-World Example

Consider an AI PR Reviewer.

Instead of sending only the changed code, ContextBuilder may assemble:

- Pull request description
- Git diff
- Coding standards
- Repository architecture
- Previous review comments
- Security checklist
- Performance guidelines

The prompt remains almost unchanged, but the model now has significantly richer information to work with.

# Best Practices

- Keep PromptBuilder and ContextBuilder separate.
- Assemble context dynamically.
- Rank information by relevance.
- Monitor token consumption.
- Make context providers modular and testable.

# Key Takeaways

- Prompt Engineering and Context Engineering solve different problems.
- Context quality often matters more than prompt wording.
- ContextBuilder assembles information from multiple sources.
- Effective context must balance relevance with token limits.
- Context Engineering is the foundation of scalable AI applications.

# Discussion Notes

During our roadmap discussions, we recognized that every AI application we plan to build—including the AI Chat Platform, PR Reviewer, and AI Learning Portal—will require a reusable ContextBuilder.

Rather than allowing each application to gather context independently, AI Core will expose a shared ContextBuilder responsible for assembling all relevant information. This decision keeps business logic clean, encourages reuse, and enables future features such as semantic retrieval, memory management, and intelligent context optimization without changing application code.
